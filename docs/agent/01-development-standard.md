# 开发执行规范

## 1. 开始任务

开始前必须读取：

- `docs/agent/00-project.md`
- 本文件
- `docs/agent/02-review-standard.md`
- 当前任务明确要求读取的契约、冻结原型和迭代资料

然后检查：

- 当前分支
- HEAD
- `.ai-dev/state.json`
- 工作区修改
- 未跟踪文件

若基线、分支或工作区不符合任务前置条件，停止并报告。

不得额外创建计划文档，不得阶段性开始后续任务。

## 2. 范围控制

只修改完成当前业务目标所需的最小文件集合。

允许范围由以下内容共同确定：

- 当前任务指令
- 当前迭代 scope
- 正式契约
- 冻结 UI

默认禁止顺带重构无关模块、升级依赖、格式化全仓库或修改历史文档。

发现任务必须依赖以下变更时，先停止并报告：

- OpenAPI 或 Contract 变化
- Schema 或 Migration 变化
- 破坏性数据库操作
- 产品规则不明确
- 权限或外部服务不可用

## 3. 分层职责

### API 层

负责：

- 请求参数与响应类型
- HTTP 状态处理
- `ErrorEnvelope` 解析
- 网络异常归一化
- 请求取消和必要的超时处理

不得把原始服务端错误直接展示给用户。

### Mapper 层

负责：

- API 数据到 ViewModel 的转换
- 枚举中文化
- 时间格式化
- 技术字段隐藏
- 空值与未知值兜底
- 统计派生和展示文案

同一枚举和展示规则不得散落在多个 JSX 中。

### ViewModel

只包含 UI 渲染和交互真正需要的数据：

- 用户可读字段
- 明确状态
- 可执行操作
- loading、empty、error 等页面状态

### UI / JSX

只消费 ViewModel。

不得：

- 直接解析 ErrorEnvelope
- 直接格式化服务端枚举
- 直接输出技术字段
- 在 JSX 中堆叠业务规则
- 使用 `href="#"`
- 使用空 `onClick`
- 使用 `console.log` 模拟成功
- 创建契约不支持的虚假按钮或开关

## 4. 数据请求

必须：

- 避免相同请求重复触发
- 避免 N+1
- 页面卸载时取消未完成请求
- 防止重复提交
- 对正式写操作提供 loading、成功刷新和失败反馈

禁止高频轮询。确需轮询时必须有明确业务依据、合理间隔和停止条件。

## 5. 页面状态

所有数据页面根据实际需要覆盖：

- initial
- loading
- success
- empty
- error
- retry

错误提示必须安全、可理解，不得暴露堆栈、SQL、服务地址、内部字段或原始响应。

## 6. UI 实现

UI 任务必须：

- 以冻结原型为视觉和结构基线
- 保持全局导航和既有布局体系一致
- 支持长名称和长说明换行
- 避免横向溢出
- 在目标桌面宽度和较窄桌面宽度下可用
- 使用真实 API 数据计算统计，不硬编码业务结果

技术字段、状态和时间统一按 `00-project.md` 转换规则处理。

## 7. 数据库与契约

除非当前任务明确授权，否则禁止修改：

- OpenAPI
- Contract
- Schema
- Migration
- 数据库初始化逻辑
- 持久化卷
- package 与 lock 文件

不得为了适配 UI 擅自伪造后端能力。

## 8. 测试策略

### 所有代码任务必须执行

- 与修改范围对应的定向测试
- 相关模块全部测试
- typecheck
- 修改文件定向 lint
- production build
- `git diff --check`

不得运行无必要的全量 lint。

### UI 任务额外执行

测试至少覆盖：

- 正常加载
- loading
- empty
- error
- retry
- 真实 DOM 与交互
- 技术字段不泄露
- 导航高亮
- 无虚假按钮
- 无横向溢出
- 无 hydration error

不得仅通过源码文本搜索替代真实 DOM 测试。

### API、Contract 或 Migration 任务额外执行

仅在任务明确授权时执行对应改动，并补充：

- 契约测试
- 数据迁移兼容性
- 回滚或失败策略
- 真实 API 联调

## 9. 命令规范

Windows 下统一使用：

```powershell
pnpm.cmd ...
```

不得通过修改系统 Execution Policy 解决 `pnpm.ps1` 问题。

测试或构建失败时：

1. 读取完整日志
2. 定位根因
3. 先运行最小局部验证
4. 修复后再运行完整门禁

不得无分析地反复执行完整门禁。

## 10. 停止条件

以下情况必须停止并报告：

- 基线不一致
- 存在无法解释的工作区修改
- 需要超范围修改
- 正式契约与产品要求冲突
- 需要破坏性数据库操作
- 权限或基础环境阻塞
- 无法完成真实验证

不得将未验证修改描述为完成。
