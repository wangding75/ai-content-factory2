# P0 非功能需求

## NFR-01 可靠性

- 核心写操作使用事务。
- WorkflowRun 与领域结果必须一致或可补偿。
- 失败分支不允许部分写入。
- 数据库迁移可在空库重复执行。

## NFR-02 性能

- 本地开发环境下普通列表和详情 API 的 P95 目标小于 500ms，不含人为模拟延时。
- 列表接口必须支持明确上限，禁止无界查询。
- 大正文不得在无需求的聚合接口中重复返回。

## NFR-03 安全

- 不将密钥写入仓库、日志或 API 响应。
- 输入进行结构、长度和枚举校验。
- SQL 使用参数化查询。
- 富文本按纯文本或安全渲染策略处理。
- P0 虽无登录，仍保留 actor_id 和未来授权边界。

## NFR-04 可观测性

- 每个 HTTP 请求生成 request_id。
- 结构化日志包含方法、路径、状态码、耗时和 request_id。
- WorkflowRun 记录输入摘要、输出摘要、错误和耗时。
- 核心写操作生成 AuditLog。

## NFR-05 可维护性

- Domain 不依赖 HTTP、PostgreSQL、Redis 或 Provider 实现。
- 契约优先，OpenAPI 和 JSON Schema 为类型来源。
- 模块化单体禁止跨模块直接访问对方数据库 Row。
- 任何架构偏离必须有 ADR。

## NFR-06 可测试性

- 每个核心需求至少有一个可重复执行验收用例。
- Repository 使用真实 PostgreSQL 集成测试。
- Web E2E 验证数据变化而非只验证页面打开。
