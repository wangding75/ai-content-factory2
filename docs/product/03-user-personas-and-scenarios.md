# AI Content Factory 2.0｜用户角色与核心场景

## 1. 核心 Persona

### Persona A：独立小说创作者

- 目标：持续完成小说项目，减少重复整理和上下文切换。
- 痛点：人物设定散落、章节计划频繁修改、AI 输出不可追踪。
- 成功标准：可以在一个项目中看到策划、素材、故事线、章节、正文和审核链路。

### Persona B：内容策划型创作者

- 目标：先结构化策划，再分阶段生成内容。
- 痛点：工具常直接生成最终文本，缺少候选、确认和版本机制。
- 成功标准：可选择候选、编辑、确认，并明确控制何时进入正文生产。

### Persona C：高级自动化用户（后续）

- 目标：连接模型和外部工作流，批量生产多种内容。
- P0 需求：只需要看到能力边界和未配置状态，不需要真实接入。

## 2. Job To Be Done

| 场景 | 用户任务 | 产品结果 |
|---|---|---|
| 启动新作品 | 快速建立小说项目与基础策划 | 生成可持续维护的项目上下文 |
| 整理设定 | 创建或复用人物、地点和世界观 | 全局资产与项目用途可区分 |
| 设计叙事 | 组织主线、子线和伏笔 | 叙事结构可视、可引用 |
| 拆解章节 | 生成候选并人工确认 | 规划可控，不直接生产正文 |
| 生产正文 | 生成或人工编辑章节 | 正文形成明确版本 |
| 质量检查 | 查看结构化审核问题和建议 | 审核结果独立、可追踪 |
| 改写内容 | 基于审核创建重写版本 | 新旧版本共存，不覆盖原稿 |

## 3. 关键使用场景

### 场景 1：从零创建小说项目

前置：用户位于首页。

结果：项目被持久化，进入项目概览，返回项目列表仍可看到该项目。

### 场景 2：项目内新建人物并在其他项目复用

前置：当前项目已存在。

结果：人物本体只创建一次；不同项目可配置不同角色用途；解除某项目绑定不删除人物。

### 场景 3：从故事线到章节规划

前置：已有主线、子线、素材和项目策划。

结果：Mock Provider 生成候选；用户编辑并确认；确认后仍需显式进入正文生产。

### 场景 4：审核与重写

前置：已有已保存版本。

结果：审核报告关联固定版本；重写产生子版本；原版本和审核报告继续可查询。

## 4. 失败场景

- 创建项目名称为空：阻止提交，数据库不写入。
- 重复绑定素材：返回可识别冲突，不生成重复 Usage。
- 编辑已确认章节计划：拒绝并返回状态冲突。
- 对未保存正文审核：拒绝并提示前置条件。
- Mock 工作流失败：WorkflowRun 标记 failed，领域结果不应部分写入。
